Reporting Engineering · End-to-End Pipeline

TL;DR — A production-style pipeline that ingests real CMS Open Payments data (pharmaceutical company payments to physicians), applies cleaning and business logic, and produces publication-quality reports — Quarto HTML and formatted Excel — with no manual steps after the first run. This is the engineering behind the methodology paper made concrete.

Data source: CMS Open Payments — public dataset of financial relationships between drug/device companies and physicians. Released by the Centers for Medicare & Medicaid Services as required by the Affordable Care Act. Multi-GB per program year, available as CSV. Public domain. The fetch script (data/fetch_cms.py) downloads the bulk file, filters to eight pharmaceutical manufacturers using Polars lazy scan, and caches the result as Parquet — bringing a 3 GB CSV down to a tens-of-MB working set.

Note: This case study is written from the perspective of an R-primary workflow because Quarto + gt + openxlsx2 is genuinely R’s strongest territory. The principles — not the specific syntax — are what transfer.

The Business Problem

Recurring reports are where analytics teams quietly hemorrhage hours. A monthly client deliverable that takes 6 hours to assemble, times 12 months, times multiple clients, adds up to enormous opportunity cost.

The CMS Open Payments dataset is a useful proving ground because it has the qualities that make real client work hard:

  • Real volume (millions of records per year)
  • Real messiness (free-text purpose fields, inconsistent vendor naming, missing values)
  • Real reporting needs (companies and analysts care about this — there’s a publication audience)

The goal: a single command produces a publication-quality monthly summary report (HTML and Excel) for any pharmaceutical company in the dataset, ready to send to a stakeholder.

The Approach

1

Modular cleaning functions

Cleaning logic lives in standalone R functions, not in the report itself. Each function does one thing and is independently testable.

2

Parameterized Quarto report

The report (monthly-summary.qmd) takes parameters: company name, reporting period. One template renders for any company.

3

Publication-quality tables (gt)

Use gt and gtExtras for tables that look like they belong in a Wall Street Journal feature, not a Jupyter notebook.

4

Excel output via openxlsx2

Same data, formatted Excel workbook with frozen panes, conditional formatting, and multiple sheets — for stakeholders who live in Excel.

Step 1: Load CMS Open Payments data

The fetch script (data/fetch_cms.py) downloads the General Payments bulk file from download.cms.gov/openpayments/, filters it to the eight target manufacturers using a Polars lazy scan, and writes the result to Parquet. CMS rotates the bulk filename on every refresh, so the script accepts the current URL as --url — see the README for instructions.

If the cache is absent (e.g. you cloned the repo and haven’t fetched yet), the chunk falls back to a clearly-labeled synthetic dataset that mirrors the real schema, so the rest of the case study still renders. The synthetic path is only there as a fallback — the goal is real data.

Step 2: Modular cleaning functions

The cleaning lives in standalone functions, in a separate R/ directory in production. Each function is testable and reusable.

Modular cleaning functions (would normally live in R/clean.R) 
#' Standardize CMS column names
#' @param df Raw CMS payments tibble
clean_column_names <- function(df) {
  df %>%
    rename(
      company       = applicable_manufacturer_or_applicable_gpo_making_payment_name,
      specialty     = physician_specialty,
      state         = recipient_state,
      payment_date  = date_of_payment,
      payment_type  = nature_of_payment_or_transfer_of_value,
      amount_usd    = total_amount_of_payment_usdollars
    )
}

#' Remove duplicate filings (keeping first occurrence)
deduplicate_records <- function(df) {
  df %>% distinct(record_id, .keep_all = TRUE)
}

#' Handle missing values with documented defaults
handle_missing <- function(df) {
  df %>%
    mutate(
      state = if_else(is.na(state), "Unknown", state),
      specialty = if_else(is.na(specialty), "Not Reported", specialty)
    )
}

#' Flag outliers using IQR method (don't remove, just flag for review)
flag_outliers <- function(df, multiplier = 3) {
  df %>%
    group_by(payment_type) %>%
    mutate(
      .q1 = quantile(amount_usd, 0.25, na.rm = TRUE),
      .q3 = quantile(amount_usd, 0.75, na.rm = TRUE),
      .iqr = .q3 - .q1,
      is_outlier = amount_usd > .q3 + multiplier * .iqr
    ) %>%
    ungroup() %>%
    select(-.q1, -.q3, -.iqr)
}

#' Add derived time dimensions
add_time_dimensions <- function(df) {
  df %>%
    mutate(
      year = year(payment_date),
      quarter = paste0("Q", quarter(payment_date)),
      month = month(payment_date, label = TRUE, abbr = TRUE),
      year_month = floor_date(payment_date, "month")
    )
}

#' Full cleaning pipeline
clean_pipeline <- function(df) {
  df %>%
    clean_column_names() %>%
    deduplicate_records() %>%
    handle_missing() %>%
    flag_outliers() %>%
    add_time_dimensions()
}

# Apply
payments <- clean_pipeline(payments_raw)

Step 3: Publication-quality tables with gt

This is where R earns its keep. gt and gtExtras produce tables that genuinely belong in a published report.

Build publication-quality summary table with gt 
library(gt)
library(scales)

# Pick the top-spend manufacturer in the cleaned data as the parameterized example.
# Real CMS company names vary slightly across program years (e.g. "Pfizer, Inc." vs
# "PFIZER INC."), so we resolve at run time rather than hard-coding a string.
target_company <- payments %>%
  filter(!is_outlier) %>%
  group_by(company) %>%
  summarise(spend = sum(amount_usd, na.rm = TRUE), .groups = "drop") %>%
  arrange(desc(spend)) %>%
  slice(1) %>%
  pull(company)

company_summary <- payments %>%
  filter(company == target_company, !is_outlier) %>%
  group_by(payment_type) %>%
  summarise(
    n_payments = n(),
    total_amount = sum(amount_usd),
    median_amount = median(amount_usd),
    avg_amount = mean(amount_usd),
    .groups = "drop"
  ) %>%
  arrange(desc(total_amount)) %>%
  mutate(
    pct_of_total = total_amount / sum(total_amount)
  )

summary_year <- year(min(payments$payment_date, na.rm = TRUE))
company_summary %>%
  gt() %>%
  tab_header(
    title = md(paste0("**", target_company, " — Payment Summary**")),
    subtitle = paste0("Open Payments General Payments · Program Year ", summary_year)
  ) %>%
  fmt_currency(columns = c(total_amount, median_amount, avg_amount), decimals = 0) %>%
  fmt_number(columns = n_payments, decimals = 0, sep_mark = ",") %>%
  fmt_percent(columns = pct_of_total, decimals = 1) %>%
  cols_label(
    payment_type = "Payment Type",
    n_payments = "Count",
    total_amount = "Total",
    median_amount = "Median",
    avg_amount = "Average",
    pct_of_total = "% of Total"
  ) %>%
  tab_style(
    style = list(cell_text(weight = "bold")),
    locations = cells_body(columns = total_amount, rows = pct_of_total > 0.15)
  ) %>%
  data_color(
    columns = pct_of_total,
    palette = c("#f8fafc", "#dbeafe", "#2563eb"),
    domain = c(0, max(company_summary$pct_of_total))
  ) %>%
  tab_source_note(source_note = "Source: CMS Open Payments · Auto-generated from pipeline") %>%
  tab_options(
    table.font.size = px(12),
    heading.title.font.size = px(15),
    column_labels.font.weight = "bold",
    table.border.top.style = "none",
    table_body.hlines.style = "none"
  )

Step 6: The runner script

In production, all of the above lives in functions called from a parameterized runner. A single command produces the full deliverable for any company:

# What the team actually runs each month:
quarto render reports/monthly-summary.qmd \
  -P company:"Pfizer Inc." \
  -P period:"2024" \
  --output-dir output/pfizer-2024/

Or programmatically, looping over all companies:

# Render reports for all companies in one batch
companies_to_render <- unique(payments$company)

purrr::walk(companies_to_render, function(co) {
  quarto::quarto_render(
    "reports/monthly-summary.qmd",
    execute_params = list(company = co, period = "2024"),
    output_file = paste0("output/", janitor::make_clean_names(co), "_2024.html")
  )
})

What this demonstrates

R fluency where R is the right answer. Quarto + gt + ggplot2 + openxlsx2 is a genuinely first-class reporting stack. This case study shows them working together as they would in production.

Modular code organization. Cleaning functions separated from report rendering. Each function has one job and could be unit-tested. This is the structure that scales beyond a single analyst’s brain.

Honest about Excel. Some stakeholders need formatted Excel. Pretending they don’t costs you credibility. Generating it from the same source data, alongside the HTML report, is the right answer.

The methodology paper, made concrete. Everything here is the practical implementation of The Case for Code-Driven Reporting — the same discipline applied to real reporting work.

What I’d extend in production

  • Orchestration with targets for dependency tracking and incremental rebuilds
  • Tests for the cleaning functions using testthat — would catch upstream schema changes the day they happen
  • Renv lockfile for fully reproducible dependencies across the team
  • CI/CD — auto-render reports on data refresh via GitHub Actions
  • Email/Slack delivery — completed reports auto-distributed to stakeholders

```{r}
      names(payment_types), n_records, replace = TRUE,
      prob = unname(payment_types)
    )
  ) %>%
    mutate(
      total_amount_of_payment_usdollars = round(rlnorm(n(), meanlog = 4.8, sdlog = 1.6), 2),
      recipient_state = ifelse(runif(n()) < 0.015, NA_character_, recipient_state),
      physician_specialty = ifelse(runif(n()) < 0.01, NA_character_, physician_specialty)
    )
  dupe_rows <- payments_raw %>% slice_sample(n = 150)
  payments_raw <- bind_rows(payments_raw, dupe_rows)
}

cat("Records: ", format(nrow(payments_raw), big.mark = ","), "\n")
cat("Total payments: ", scales::dollar(sum(payments_raw$total_amount_of_payment_usdollars)), "\n")
cat("Unique companies: ", length(unique(payments_raw$applicable_manufacturer_or_applicable_gpo_making_payment_name)), "\n")
cat("Date range: ", as.character(min(payments_raw$date_of_payment)),
    " to ", as.character(max(payments_raw$date_of_payment)), "\n", sep = "")
```

## Step 2: Modular cleaning functions

The cleaning lives in standalone functions, in a separate `R/` directory in production.
Each function is testable and reusable.

```{r}
#| label: cleaning-functions
#| code-summary: "Modular cleaning functions (would normally live in R/clean.R)"

#' Standardize CMS column names
#' @param df Raw CMS payments tibble
clean_column_names <- function(df) {
  df %>%
    rename(
      company       = applicable_manufacturer_or_applicable_gpo_making_payment_name,
      specialty     = physician_specialty,
      state         = recipient_state,
      payment_date  = date_of_payment,
      payment_type  = nature_of_payment_or_transfer_of_value,
      amount_usd    = total_amount_of_payment_usdollars
    )
}

#' Remove duplicate filings (keeping first occurrence)
deduplicate_records <- function(df) {
  df %>% distinct(record_id, .keep_all = TRUE)
}

#' Handle missing values with documented defaults
handle_missing <- function(df) {
  df %>%
    mutate(
      state = if_else(is.na(state), "Unknown", state),
      specialty = if_else(is.na(specialty), "Not Reported", specialty)
    )
}

#' Flag outliers using IQR method (don't remove, just flag for review)
flag_outliers <- function(df, multiplier = 3) {
  df %>%
    group_by(payment_type) %>%
    mutate(
      .q1 = quantile(amount_usd, 0.25, na.rm = TRUE),
      .q3 = quantile(amount_usd, 0.75, na.rm = TRUE),
      .iqr = .q3 - .q1,
      is_outlier = amount_usd > .q3 + multiplier * .iqr
    ) %>%
    ungroup() %>%
    select(-.q1, -.q3, -.iqr)
}

#' Add derived time dimensions
add_time_dimensions <- function(df) {
  df %>%
    mutate(
      year = year(payment_date),
      quarter = paste0("Q", quarter(payment_date)),
      month = month(payment_date, label = TRUE, abbr = TRUE),
      year_month = floor_date(payment_date, "month")
    )
}

#' Full cleaning pipeline
clean_pipeline <- function(df) {
  df %>%
    clean_column_names() %>%
    deduplicate_records() %>%
    handle_missing() %>%
    flag_outliers() %>%
    add_time_dimensions()
}

# Apply
payments <- clean_pipeline(payments_raw)

cat("After cleaning:\n")
cat("  Records:    ", format(nrow(payments), big.mark = ","), "\n")
cat("  Outliers:   ", format(sum(payments$is_outlier), big.mark = ","),
    " (flagged, not removed)\n")
cat("  Companies:  ", length(unique(payments$company)), "\n")
```

## Step 3: Publication-quality tables with `gt`

This is where R earns its keep. `gt` and `gtExtras` produce tables that genuinely
belong in a published report.

```{r}
#| label: gt-table
#| code-summary: "Build publication-quality summary table with gt"
#| message: false

library(gt)
library(scales)

# Pick the top-spend manufacturer in the cleaned data as the parameterized example.
# Real CMS company names vary slightly across program years (e.g. "Pfizer, Inc." vs
# "PFIZER INC."), so we resolve at run time rather than hard-coding a string.
target_company <- payments %>%
  filter(!is_outlier) %>%
  group_by(company) %>%
  summarise(spend = sum(amount_usd, na.rm = TRUE), .groups = "drop") %>%
  arrange(desc(spend)) %>%
  slice(1) %>%
  pull(company)

company_summary <- payments %>%
  filter(company == target_company, !is_outlier) %>%
  group_by(payment_type) %>%
  summarise(
    n_payments = n(),
    total_amount = sum(amount_usd),
    median_amount = median(amount_usd),
    avg_amount = mean(amount_usd),
    .groups = "drop"
  ) %>%
  arrange(desc(total_amount)) %>%
  mutate(
    pct_of_total = total_amount / sum(total_amount)
  )

summary_year <- year(min(payments$payment_date, na.rm = TRUE))
company_summary %>%
  gt() %>%
  tab_header(
    title = md(paste0("**", target_company, " — Payment Summary**")),
    subtitle = paste0("Open Payments General Payments · Program Year ", summary_year)
  ) %>%
  fmt_currency(columns = c(total_amount, median_amount, avg_amount), decimals = 0) %>%
  fmt_number(columns = n_payments, decimals = 0, sep_mark = ",") %>%
  fmt_percent(columns = pct_of_total, decimals = 1) %>%
  cols_label(
    payment_type = "Payment Type",
    n_payments = "Count",
    total_amount = "Total",
    median_amount = "Median",
    avg_amount = "Average",
    pct_of_total = "% of Total"
  ) %>%
  tab_style(
    style = list(cell_text(weight = "bold")),
    locations = cells_body(columns = total_amount, rows = pct_of_total > 0.15)
  ) %>%
  data_color(
    columns = pct_of_total,
    palette = c("#f8fafc", "#dbeafe", "#2563eb"),
    domain = c(0, max(company_summary$pct_of_total))
  ) %>%
  tab_source_note(source_note = "Source: CMS Open Payments · Auto-generated from pipeline") %>%
  tab_options(
    table.font.size = px(12),
    heading.title.font.size = px(15),
    column_labels.font.weight = "bold",
    table.border.top.style = "none",
    table_body.hlines.style = "none"
  )
```

## Step 6: The runner script

In production, all of the above lives in functions called from a parameterized runner.
A single command produces the full deliverable for any company:

```bash
# What the team actually runs each month:
quarto render reports/monthly-summary.qmd \
  -P company:"Pfizer Inc." \
  -P period:"2024" \
  --output-dir output/pfizer-2024/
```

Or programmatically, looping over all companies:

```r
# Render reports for all companies in one batch
companies_to_render <- unique(payments$company)

purrr::walk(companies_to_render, function(co) {
  quarto::quarto_render(
    "reports/monthly-summary.qmd",
    execute_params = list(company = co, period = "2024"),
```

## What this demonstrates

**R fluency where R is the right answer.** Quarto + gt + ggplot2 + openxlsx2 is a
genuinely first-class reporting stack. This case study shows them working together as
they would in production.

**Modular code organization.** Cleaning functions separated from report rendering. Each
function has one job and could be unit-tested. This is the structure that scales beyond
a single analyst's brain.

**Honest about Excel.** Some stakeholders need formatted Excel. Pretending they don't
costs you credibility. Generating it from the same source data, alongside the HTML report,
is the right answer.

**The methodology paper, made concrete.** Everything here is the practical implementation
of [The Case for Code-Driven Reporting](../posts/code-driven-reporting.html) — the same
discipline applied to real reporting work.

## What I'd extend in production

- **Orchestration** with [targets](https://docs.ropensci.org/targets/) for dependency
  tracking and incremental rebuilds
- **Tests** for the cleaning functions using `testthat` — would catch upstream schema
  changes the day they happen
- **Renv lockfile** for fully reproducible dependencies across the team
- **CI/CD** — auto-render reports on data refresh via GitHub Actions
- **Email/Slack delivery** — completed reports auto-distributed to stakeholders
