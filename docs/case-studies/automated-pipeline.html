Reporting Engineering · End-to-End Pipeline

TL;DR — A production style pipeline that ingests real CMS Open Payments data (pharma company payments to physicians), runs it through modular cleaning functions, and produces publication quality reports in Quarto HTML and formatted Excel with no manual steps after the first run. This is the engineering behind the methodology paper made concrete, on 3.8 million real records.

Data source: CMS Open Payments, the public dataset of financial relationships between drug and device companies and physicians, released by the Centers for Medicare & Medicaid Services under the Affordable Care Act. The bulk file runs multiple GB per program year. The fetch script (data/fetch_cms.py) downloads it, filters to eight pharmaceutical manufacturers with a Polars lazy scan, and caches the result as Parquet. A 3 GB CSV becomes a 41 MB working set.

Note: this case study is written from an R-first perspective because Quarto + gt + openxlsx2 is genuinely R’s strongest territory. The principles transfer, the syntax is just the costume.

The Business Problem

Recurring reports are where analytics teams quietly bleed hours. A monthly client deliverable that takes 6 hours to assemble, times 12 months, times multiple clients, that adds up to real money and nobody budgets for it.

The CMS Open Payments dataset is a useful proving ground because it has the same qualities that make real client work hard:

  • Real volume, millions of records per program year
  • Real messiness, the same parent company files under a pile of legal entity names, payment type labels run to 140 characters
  • Real reporting needs, journalists and compliance teams actually publish off this data

The goal: one command produces a publication quality summary report, HTML and Excel, for any manufacturer in the dataset, ready to send to a stakeholder.

The Approach

1

Modular cleaning functions

Cleaning logic lives in standalone R functions, not buried in the report. Each function does one thing and is independently testable.

2

Parameterized Quarto report

The report template takes parameters, company name and reporting period. One template renders for any company.

3

Publication quality tables (gt)

gt and gtExtras for tables that look like they belong in a Wall Street Journal feature, not a notebook dump.

4

Excel output via openxlsx2

Same data, formatted Excel workbook with frozen panes and multiple sheets, for the stakeholders who live in Excel. Which is most of them.

Step 1: Load CMS Open Payments data

The fetch script (data/fetch_cms.py) downloads the General Payments bulk file from download.cms.gov/openpayments/, filters it to the eight target manufacturers with a Polars lazy scan, and writes Parquet. CMS rotates the bulk filename on every refresh so the script takes the current URL as --url, the README has the details.

If the cache is missing (you cloned the repo and haven’t fetched yet) the chunk falls back to a clearly labeled synthetic dataset with the same schema so the page still renders. The synthetic path is only a fallback, the goal is real data and everything below was rendered against it.

Step 2: Modular cleaning functions

The cleaning lives in standalone functions, in a separate R/ directory in production. Each one is testable on its own and the report just calls the pipeline.

The one worth a second look is roll_up_company(). The raw file lists 26 different payer names for what a stakeholder would call 7 companies. “PFIZER INC.” and “PFIZER INTERNATIONAL LLC” are not two companies, and Janssen is just Johnson & Johnson wearing a different badge. Skip this step and every company level number in the report is wrong.

Modular cleaning functions (would live in R/clean.R in production) 
## rename the wall-of-text CMS bulk columns to names a human can type
clean_column_names <- function(df) {
  df %>%
    rename(
      company       = applicable_manufacturer_or_applicable_gpo_making_payment_name,
      specialty     = physician_specialty,
      state         = recipient_state,
      payment_date  = date_of_payment,
      payment_type  = nature_of_payment_or_transfer_of_value,
      amount_usd    = total_amount_of_payment_usdollars
    )
}

## drop duplicate filings, keep the first occurrence
deduplicate_records <- function(df) {
  df %>% distinct(record_id, .keep_all = TRUE)
}

## missing values get documented defaults instead of silent NA drops
handle_missing <- function(df) {
  df %>%
    mutate(
      state = if_else(is.na(state), "Unknown", state),
      specialty = if_else(is.na(specialty), "Not Reported", specialty)
    )
}

## the same parent files under a pile of legal entities. roll them up so
## company level numbers mean what a reader thinks they mean
### same entity resolution problem as the FEC vendor names in the political
### spend study, just smaller. 26 names here, so a checked-by-hand regex
### map is enough. at vendor scale you graduate to fuzzy matching
roll_up_company <- function(df) {
  df %>%
    mutate(parent = case_when(
      str_detect(str_to_upper(company), "ABBVIE")          ~ "AbbVie",
      str_detect(str_to_upper(company), "PFIZER")          ~ "Pfizer",
      str_detect(str_to_upper(company), "NOVARTIS")        ~ "Novartis",
      str_detect(str_to_upper(company), "GLAXO|GSK")       ~ "GSK",
      str_detect(str_to_upper(company), "JANSSEN|JOHNSON") ~ "Johnson & Johnson",
      str_detect(str_to_upper(company), "MERCK")           ~ "Merck",
      str_detect(str_to_upper(company), "LILLY")           ~ "Eli Lilly",
      str_detect(str_to_upper(company), "BRISTOL")         ~ "Bristol Myers Squibb",
      TRUE ~ company   ## anything unmapped keeps its raw name so it's visible
    ))
}

## CMS payment type labels run up to 140 characters. nobody wants that in
## a table row, and the longest one is also the biggest dollar bucket
shorten_payment_types <- function(df) {
  df %>%
    mutate(payment_type_short = case_when(
      str_detect(payment_type, "^Compensation for services other than consulting") ~ "Speaker / services fees",
      str_detect(payment_type, "^Compensation for serving as faculty")             ~ "CME speaker fees",
      payment_type == "Food and Beverage"   ~ "Meals",
      payment_type == "Consulting Fee"      ~ "Consulting",
      payment_type == "Travel and Lodging"  ~ "Travel and lodging",
      payment_type == "Royalty or License"  ~ "Royalties",
      str_detect(payment_type, "^Space rental")            ~ "Facility fees",
      str_detect(payment_type, "^Long term medical supply") ~ "Device loans",
      TRUE ~ payment_type
    ))
}

## flag outliers with 3x IQR inside each payment type. flag, never remove.
### a $22,000 royalty is normal, a $22,000 meal is not, that's why the
### threshold is computed per payment type and not globally
flag_outliers <- function(df, multiplier = 3) {
  df %>%
    group_by(payment_type) %>%
    mutate(
      .q1 = quantile(amount_usd, 0.25, na.rm = TRUE),
      .q3 = quantile(amount_usd, 0.75, na.rm = TRUE),
      is_outlier = amount_usd > .q3 + multiplier * (.q3 - .q1)
    ) %>%
    ungroup() %>%
    select(-.q1, -.q3)
}

## derived time dimensions, every report downstream wants these
add_time_dimensions <- function(df) {
  df %>%
    mutate(
      year = year(payment_date),
      quarter = paste0("Q", quarter(payment_date)),
      month = month(payment_date, label = TRUE, abbr = TRUE),
      year_month = floor_date(payment_date, "month")
    )
}

## the full pipeline reads like the run sheet
clean_pipeline <- function(df) {
  df %>%
    clean_column_names() %>%
    deduplicate_records() %>%
    handle_missing() %>%
    roll_up_company() %>%
    shorten_payment_types() %>%
    flag_outliers() %>%
    add_time_dimensions()
}

payments <- clean_pipeline(payments_raw)

Step 3: Publication quality tables with gt

This is where R earns its keep. gt produces tables that genuinely belong in a published report. The summary excludes flagged outliers on purpose, they get their own sheet in the Excel workbook later so nothing disappears, it just gets labeled.

Build publication-quality summary table with gt 
library(gt)
library(scales)

## top spend parent in the cleaned data becomes the parameterized example.
## resolved at run time, not hard coded, because the ranking is allowed to
## change when the data refreshes
target_company <- payments %>%
  filter(!is_outlier) %>%
  count(parent, wt = amount_usd, name = "spend") %>%
  slice_max(spend, n = 1) %>%
  pull(parent)

company_summary <- payments %>%
  filter(parent == target_company, !is_outlier) %>%
  group_by(payment_type_short) %>%
  summarise(
    n_payments = n(),
    total_amount = sum(amount_usd),
    median_amount = median(amount_usd),
    avg_amount = mean(amount_usd),
    .groups = "drop"
  ) %>%
  arrange(desc(total_amount)) %>%
  mutate(pct_of_total = total_amount / sum(total_amount))

summary_year <- year(min(payments$payment_date, na.rm = TRUE))
company_summary %>%
  gt() %>%
  tab_header(
    title = md(paste0("**", target_company, " — Payment Summary**")),
    subtitle = paste0("Open Payments General Payments · Program Year ", summary_year,
                      " · outliers excluded")
  ) %>%
  fmt_currency(columns = c(total_amount, median_amount, avg_amount), decimals = 0) %>%
  fmt_number(columns = n_payments, decimals = 0, sep_mark = ",") %>%
  fmt_percent(columns = pct_of_total, decimals = 1) %>%
  cols_label(
    payment_type_short = "Payment Type",
    n_payments = "Count",
    total_amount = "Total",
    median_amount = "Median",
    avg_amount = "Average",
    pct_of_total = "% of Total"
  ) %>%
  tab_style(
    style = list(cell_text(weight = "bold")),
    locations = cells_body(columns = total_amount, rows = pct_of_total > 0.15)
  ) %>%
  data_color(
    columns = pct_of_total,
    palette = c("#f8fafc", "#dbeafe", "#2563eb"),
    domain = c(0, max(company_summary$pct_of_total))
  ) %>%
  tab_source_note(source_note = "Source: CMS Open Payments · Auto-generated from pipeline") %>%
  tab_options(
    table.font.size = px(12),
    heading.title.font.size = px(15),
    column_labels.font.weight = "bold",
    table.border.top.style = "none",
    table_body.hlines.style = "none"
  )

Where the money actually goes

The table is the deliverable, but the chart is what gets remembered. Same rules as the other case studies: the title states the finding, labels sit on the data instead of a legend, one idea per chart.

And the finding here is a good one. Count the payments and this dataset is a catering ledger, nearly every record is a meal with an $18 median. Count the dollars and the meals barely matter, the money is speaker fees and consulting.

Shared chart style 
library(ggplot2)

## same palette as the rest of the portfolio. navy carries the money story,

    theme(

      plot.subtitle = element_text(color = GRAY, size = 9.5),
      plot.caption = element_text(color = GRAY, size = 8.5),
      plot.title.position = "plot",

Share of payment count vs share of dollars by payment type 
## five buckets tell the story, everything else is rounding error
mix <- payments %>%
  mutate(bucket = case_when(
    payment_type_short %in% c("Speaker / services fees", "CME speaker fees") ~ "Speaker fees",
    payment_type_short %in% c("Meals", "Consulting", "Travel and lodging")   ~ payment_type_short,
    TRUE ~ "All other"
  )) %>%
  group_by(bucket) %>%
  summarise(n = n(), dollars = sum(amount_usd), .groups = "drop") %>%
  mutate(
    pct_n = n / sum(n) * 100,
    pct_d = dollars / sum(dollars) * 100
  )

## order by dollars so the money story reads top down
mix <- mix %>% arrange(pct_d)
mix$bucket <- factor(mix$bucket, levels = mix$bucket)

long <- mix %>%
  pivot_longer(c(pct_d, pct_n), names_to = "measure", values_to = "pct") %>%
  mutate(measure = factor(measure, levels = c("pct_n", "pct_d")))

### the top group's labels name the series, so no legend box needed
top_bucket <- levels(mix$bucket)[nrow(mix)]
long <- long %>%
  mutate(lab = case_when(
    bucket == top_bucket & measure == "pct_d" ~ sprintf("%.0f%% of the dollars", pct),
    bucket == top_bucket & measure == "pct_n" ~ sprintf("%.0f%% of the payments", pct),
    pct < 1   ~ sprintf("%.1f%%", pct),
    TRUE      ~ sprintf("%.0f%%", pct)
  ))

ggplot(long, aes(x = pct, y = bucket, fill = measure)) +
  geom_col(position = position_dodge(width = 0.72), width = 0.62) +
  geom_text(aes(label = lab, color = measure),
            position = position_dodge(width = 0.72),
            hjust = -0.12, size = 3.4, fontface = "bold") +
  scale_fill_manual(values = c(pct_d = NAVY, pct_n = SLATE)) +
  scale_color_manual(values = c(pct_d = NAVY, pct_n = SLATE)) +
  scale_x_continuous(limits = c(0, 118), expand = c(0, 0)) +
  labs(
    title = "Almost every payment is a meal. Almost none of the money is",
    subtitle = sprintf("share of record count vs share of dollars by payment type, CMS program year %d, %d manufacturers", summary_year, n_parents),
    x = NULL, y = NULL

This is the chart a compliance officer or a journalist actually needs. Headlines about pharma money usually count payments, which means they’re mostly counting sandwiches. The dollars sit somewhere else entirely.

One company is half the money, every month

Roll the 26 filing names up to parents (Step 2) and the monthly cadence gets simple. Without the rollup this chart is a lie, AbbVie’s lead shrinks and Johnson & Johnson splits into five entities that all look mid-pack.

Monthly payment volume by parent company 
top4 <- payments %>%
  filter(!is_outlier) %>%
  count(parent, wt = amount_usd, name = "spend") %>%
  slice_max(spend, n = 4) %>%
  pull(parent)

monthly_trend <- payments %>%
  filter(!is_outlier, parent %in% top4) %>%
  group_by(year_month, parent) %>%
  summarise(total = sum(amount_usd) / 1e6, .groups = "drop")

trend_palette <- setNames(c(NAVY, BLUE, GREEN, AMBER), top4)

## abbvie's share of ALL dollars, the headline reads this
abbvie_share <- payments %>%
  summarise(s = sum(amount_usd[parent == top4[1]]) / sum(amount_usd) * 100) %>%
  pull(s)

## company names ride the end of their own lines. the bottom two finish the
## year close together so they each get a nudge, otherwise the labels collide
ends <- monthly_trend %>%
  group_by(parent) %>%
  slice_max(year_month, n = 1) %>%
  ungroup() %>%
  arrange(total) %>%
  mutate(label_y = total + c(-0.55, 0.55, 0, 0))

ggplot(monthly_trend, aes(year_month, total, color = parent)) +
  geom_line(linewidth = 1.1) +
  geom_point(size = 2) +
  geom_text(data = ends, aes(y = label_y, label = parent),
            hjust = 0, nudge_x = 9, size = 3.5, fontface = "bold") +

  scale_y_continuous(labels = label_dollar(suffix = "M")) +

    title = sprintf("%s alone is %.0f%% of all the %d money", top4[1], abbvie_share, summary_year),

Step 4: Render to Excel via openxlsx2

Some stakeholders need formatted Excel and pretending otherwise just costs you credibility. Generate it from the same cleaned data, alongside the HTML, and everyone gets the format they actually open.

Generate formatted Excel workbook

### in the real pipeline:

What this demonstrates

R fluency where R is the right answer. Quarto + gt + ggplot2 + openxlsx2 is a genuinely first class reporting stack, and this shows the pieces working together the way they would in production.

Entity resolution is everywhere. 26 raw payer names collapse to 7 parent companies, and skipping that step changes the answer to every company level question on this page. The rollup has rough edges worth admitting, the regex map is maintained by hand, so the day CMS adds a new subsidiary it lands unmapped until someone notices. The unmapped names keep their raw form on purpose so they show up instead of vanishing into an Other bucket. Production wants a reviewed mapping table, not a smarter regex.

Flag, don’t delete. Outliers get flagged inside each payment type and carried through to their own Excel sheet. A $22,000 royalty is normal and a $22,000 meal is not, a global threshold can’t tell those apart.

The methodology paper, made concrete. Everything here is the practical implementation of The Case for Code-Driven Reporting, the same discipline applied to a real recurring deliverable.

What I’d extend in production

  • Orchestration with targets for dependency tracking and incremental rebuilds
  • Tests for the cleaning functions with testthat, which would catch upstream schema changes the day they happen, and a unit test on the company rollup map
  • renv lockfile so the whole team renders against identical dependencies
  • CI/CD, auto render on data refresh via GitHub Actions
  • Email and Slack delivery, finished reports distributed without anyone asking

## Step 1: Load CMS Open Payments data

The fetch script (`data/fetch_cms.py`) downloads the General Payments bulk file from
`download.cms.gov/openpayments/`, filters it to the eight target manufacturers with a
Polars lazy scan, and writes Parquet. CMS rotates the bulk filename on every refresh so
the script takes the current URL as `--url`, the README has the details.

If the cache is missing (you cloned the repo and haven't fetched yet) the chunk falls
back to a clearly labeled synthetic dataset with the same schema so the page still
renders. The synthetic path is only a fallback, the goal is real data and everything
below was rendered against it.

## Step 2: Modular cleaning functions

The cleaning lives in standalone functions, in a separate `R/` directory in production.
Each one is testable on its own and the report just calls the pipeline.

The one worth a second look is `roll_up_company()`. The raw file lists 26 different
payer names for what a stakeholder would call 7 companies. "PFIZER INC." and "PFIZER
INTERNATIONAL LLC" are not two companies, and Janssen is just Johnson & Johnson wearing
a different badge. Skip this step and every company level number in the report is wrong.

```{r}
#| label: cleaning-functions
#| code-summary: "Modular cleaning functions (would live in R/clean.R in production)"

## rename the wall-of-text CMS bulk columns to names a human can type
clean_column_names <- function(df) {
  df %>%
    rename(
      company       = applicable_manufacturer_or_applicable_gpo_making_payment_name,
      specialty     = physician_specialty,
      state         = recipient_state,
      payment_date  = date_of_payment,
      payment_type  = nature_of_payment_or_transfer_of_value,
      amount_usd    = total_amount_of_payment_usdollars
    )
}

## drop duplicate filings, keep the first occurrence
deduplicate_records <- function(df) {
  df %>% distinct(record_id, .keep_all = TRUE)
}

## missing values get documented defaults instead of silent NA drops
handle_missing <- function(df) {
  df %>%
    mutate(
      state = if_else(is.na(state), "Unknown", state),
      specialty = if_else(is.na(specialty), "Not Reported", specialty)
    )
}

## the same parent files under a pile of legal entities. roll them up so
## company level numbers mean what a reader thinks they mean
### same entity resolution problem as the FEC vendor names in the political
### spend study, just smaller. 26 names here, so a checked-by-hand regex
### map is enough. at vendor scale you graduate to fuzzy matching
roll_up_company <- function(df) {
  df %>%
    mutate(parent = case_when(
      str_detect(str_to_upper(company), "ABBVIE")          ~ "AbbVie",
      str_detect(str_to_upper(company), "PFIZER")          ~ "Pfizer",
      str_detect(str_to_upper(company), "NOVARTIS")        ~ "Novartis",
      str_detect(str_to_upper(company), "GLAXO|GSK")       ~ "GSK",
      str_detect(str_to_upper(company), "JANSSEN|JOHNSON") ~ "Johnson & Johnson",
      str_detect(str_to_upper(company), "MERCK")           ~ "Merck",
      str_detect(str_to_upper(company), "LILLY")           ~ "Eli Lilly",
      str_detect(str_to_upper(company), "BRISTOL")         ~ "Bristol Myers Squibb",
      TRUE ~ company   ## anything unmapped keeps its raw name so it's visible
    ))
}

## CMS payment type labels run up to 140 characters. nobody wants that in
## a table row, and the longest one is also the biggest dollar bucket
shorten_payment_types <- function(df) {
  df %>%
    mutate(payment_type_short = case_when(
      str_detect(payment_type, "^Compensation for services other than consulting") ~ "Speaker / services fees",
      str_detect(payment_type, "^Compensation for serving as faculty")             ~ "CME speaker fees",
      payment_type == "Food and Beverage"   ~ "Meals",
      payment_type == "Consulting Fee"      ~ "Consulting",
      payment_type == "Travel and Lodging"  ~ "Travel and lodging",
      payment_type == "Royalty or License"  ~ "Royalties",
      str_detect(payment_type, "^Space rental")            ~ "Facility fees",
      str_detect(payment_type, "^Long term medical supply") ~ "Device loans",
      TRUE ~ payment_type
    ))
}

## flag outliers with 3x IQR inside each payment type. flag, never remove.
### a $22,000 royalty is normal, a $22,000 meal is not, that's why the
### threshold is computed per payment type and not globally
flag_outliers <- function(df, multiplier = 3) {
  df %>%
    group_by(payment_type) %>%
    mutate(
      .q1 = quantile(amount_usd, 0.25, na.rm = TRUE),
      .q3 = quantile(amount_usd, 0.75, na.rm = TRUE),
      is_outlier = amount_usd > .q3 + multiplier * (.q3 - .q1)
    ) %>%
    ungroup() %>%
    select(-.q1, -.q3)
}

## derived time dimensions, every report downstream wants these
add_time_dimensions <- function(df) {
  df %>%
    mutate(
      year = year(payment_date),
      quarter = paste0("Q", quarter(payment_date)),
      month = month(payment_date, label = TRUE, abbr = TRUE),
      year_month = floor_date(payment_date, "month")
    )
}

## the full pipeline reads like the run sheet
clean_pipeline <- function(df) {
  df %>%
    clean_column_names() %>%
    deduplicate_records() %>%
    handle_missing() %>%
    roll_up_company() %>%
    shorten_payment_types() %>%
    flag_outliers() %>%
    add_time_dimensions()
}

payments <- clean_pipeline(payments_raw)

## the key results section reads these at render time, no hand-typed numbers
n_raw_names <- n_distinct(payments$company)
n_parents   <- n_distinct(payments$parent)
n_dupes     <- nrow(payments_raw) - nrow(payments)
pct_outliers <- mean(payments$is_outlier) * 100

fmt_n <- function(n) if (n >= 1e6) sprintf("%.1fM", n / 1e6) else format(n, big.mark = ",")

cat("after cleaning:\n")
cat("  records:   ", format(nrow(payments), big.mark = ","), "\n")
cat("  dupes out: ", format(n_dupes, big.mark = ","), "\n")
cat("  outliers:  ", format(sum(payments$is_outlier), big.mark = ","),
    sprintf(" (%.1f%%, flagged not removed)\n", pct_outliers))
cat("  companies: ", n_raw_names, "raw filing names ->", n_parents, "parents\n")
```

## Step 3: Publication quality tables with `gt`

This is where R earns its keep. `gt` produces tables that genuinely belong in a
published report. The summary excludes flagged outliers on purpose, they get their own
sheet in the Excel workbook later so nothing disappears, it just gets labeled.

```{r}
#| label: gt-table
#| code-summary: "Build publication-quality summary table with gt"
#| message: false

library(gt)
library(scales)

## top spend parent in the cleaned data becomes the parameterized example.
## resolved at run time, not hard coded, because the ranking is allowed to
## change when the data refreshes
target_company <- payments %>%
  filter(!is_outlier) %>%
  count(parent, wt = amount_usd, name = "spend") %>%
  slice_max(spend, n = 1) %>%
  pull(parent)

company_summary <- payments %>%
  filter(parent == target_company, !is_outlier) %>%
  group_by(payment_type_short) %>%
  summarise(
    n_payments = n(),
    total_amount = sum(amount_usd),
    median_amount = median(amount_usd),
    avg_amount = mean(amount_usd),
    .groups = "drop"
  ) %>%
  arrange(desc(total_amount)) %>%
  mutate(pct_of_total = total_amount / sum(total_amount))

company_summary %>%

## Where the money actually goes

The table is the deliverable, but the chart is what gets remembered. Same rules as the
other case studies: the title states the finding, labels sit on the data instead of a
legend, one idea per chart.

ledger, nearly every record is a meal with an $18 median. Count the dollars and the
meals barely matter, the money is speaker fees and consulting.

            hjust = 0, nudge_x = 9, size = 3.5, fontface = "bold") +
  scale_color_manual(values = trend_palette) +
  scale_y_continuous(labels = label_dollar(suffix = "M")) +
  ## explicit breaks, otherwise the room reserved for the line end labels
  ## grows phantom month ticks past december
  scale_x_date(breaks = seq(min(monthly_trend$year_month),
                            max(monthly_trend$year_month), by = "month"),
               date_labels = "%b",
               expand = expansion(mult = c(0.02, 0.24))) +
  labs(
    title = sprintf("%s alone is %.0f%% of all the %d money", top4[1], abbvie_share, summary_year),
    subtitle = "monthly payments by parent company, top 4 of 7, flagged outliers excluded",
    x = NULL, y = NULL
  ) +
  theme_exec() +
  theme(panel.grid.major.x = element_blank())
```

In production all of the above lives in functions called from a parameterized runner.

  -P company:"AbbVie" \
  -P period:"2024" \

companies_to_render <- unique(payments$parent)

## What this demonstrates

**R fluency where R is the right answer.** Quarto + gt + ggplot2 + openxlsx2 is a
genuinely first class reporting stack, and this shows the pieces working together the
way they would in production.

**Entity resolution is everywhere.** `{r} n_raw_names` raw payer names collapse to
`{r} n_parents` parent companies, and skipping that step changes the answer to every
company level question on this page. The rollup has rough edges worth admitting, the
regex map is maintained by hand, so the day CMS adds a new subsidiary it lands unmapped
until someone notices. The unmapped names keep their raw form on purpose so they show up
instead of vanishing into an Other bucket. Production wants a reviewed mapping table, not
a smarter regex.

**Flag, don't delete.** Outliers get flagged inside each payment type and carried
through to their own Excel sheet. A $22,000 royalty is normal and a $22,000 meal is not,
a global threshold can't tell those apart.

**The methodology paper, made concrete.** Everything here is the practical
implementation of [The Case for Code-Driven Reporting](../posts/code-driven-reporting.html),
the same discipline applied to a real recurring deliverable.

## What I'd extend in production

- **Orchestration** with [targets](https://docs.ropensci.org/targets/) for dependency
  tracking and incremental rebuilds
- **Tests** for the cleaning functions with `testthat`, which would catch upstream schema
  changes the day they happen, and a unit test on the company rollup map
- **renv lockfile** so the whole team renders against identical dependencies
- **CI/CD**, auto render on data refresh via GitHub Actions
- **Email and Slack delivery**, finished reports distributed without anyone asking
